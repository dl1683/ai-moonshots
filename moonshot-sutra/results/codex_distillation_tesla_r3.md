First, the blocker: the active training path in [sutra_v05_train.py](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/code/sutra_v05_train.py#L47) uses `create_v052`, which still instantiates `SutraV05` in [launch_v052.py](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/code/launch_v052.py#L43). That forward explicitly says “No verify/reroute yet” in [sutra_v05_ssm.py](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/code/sutra_v05_ssm.py#L267) and returns zero compute cost in [sutra_v05_ssm.py](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/code/sutra_v05_ssm.py#L318). So real Stage-7 absorption is not a training-loop-only patch; you need model traces plus a real verify/halting path.

- **A) Encoder -> Stage 5.** Do not align raw `mu` at every step. Align a Stage-5 summary: `z5_i = Σ_s α_s,i LN(mu_post_write[s,i])`, with `α_s,i ∝ pi_5[s,i]` over recurrent steps. Teacher target is not full-sequence DeBERTa state; it is the last-token state from a prefix-only run `x[:i+1]`, preferably mean of the last 4 DeBERTa layers at position `i`. That prevents leakage exactly, because token `i` never sees tokens `> i` in the teacher input. Use a learned projector `P5` and `L_enc = mean_i m_i * (1 - cos(P5(z5_i), LN(hE_i)))`; do not use raw MSE, and do not use CKA in v1. Align only sparse anchor positions, not all positions: 2-4 anchors per sequence, biased toward low-`lambda`, high-routing, or high-loss tokens. `lambda` should stay architecture-native; teacher signal gates where Stage 5 is corrected, but does not directly write or target `lambda`. If you want calibration, add only a tiny rank regularizer so high `lambda` correlates with low final CE.

- **B) AR -> Stage 7.** If tokenizer matches, use standard temperature KL, but only on unresolved tokens: `L_ar = mean_i m_i * T^2 * KL(q_teacher^T || p_student^T)`. `p_student` should be the halt-weighted Stage-7 readout distribution across recurrent steps, not a blind last-step logit. `distilgpt2` is only worth it as a plumbing teacher; it is too weak to be the real AR knowledge source. The clean first real teacher is `gpt2` or `gpt2-medium`, not Pythia. If you insist on `pythia-160m`, do not do vocab KL. Use candidate-span KD: teacher top-K token pieces -> detokenize to text -> re-tokenize with GPT-2 BPE -> score those spans under Sutra -> KL over normalized span probabilities. That works, but it is a v2 experiment.

- **C) Training schedule.** Not simultaneous from step 0. Use `CE-first -> encoder -> dual -> CE-last`. Exact schedule: `0-30%` CE only; `30-55%` add encoder KD at `0.05`; `55-80%` add AR KD at `0.08-0.10` while keeping encoder at `0.03-0.05`; `80-100%` anneal both to `0`. Keep CE at `1.0` throughout and cap total KD mass at `<= 0.15-0.20` of the effective loss. In the overlap phase, do not let both teachers hit the same token: use disjoint masks, with encoder supervising low-`lambda` Stage-5 tokens and AR supervising high-entropy or verify-fail Stage-7 tokens. For data, do mechanism ablations on MiniPile only. After that works, scale on FineWeb-Edu; do not change corpus and teacher regime in the same experiment. The repo already has FineWeb GPT-2 tokenization wired in [tokenize_fineweb.py](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/code/tokenize_fineweb.py#L1).

- **D) Stage interaction.** In v1, the transition kernel does not explicitly route between “teacher types.” It will only react to changed internal state, `lambda`, and verify quality. That is good for the first test because teachers remain optional. Could it develop encoder mode and AR mode? Implicitly yes, but I would not wire teacher IDs into the kernel because they disappear at inference. If you want explicit modes later, add a tiny self-predicted 2-d gap head: one scalar predicts “memory belief gap,” one predicts “readout gap.” Train those heads against stop-grad encoder mismatch and AR KL, then feed the predicted gaps, not teacher outputs, into the kernel and halter.

- **E) Failure modes.** Main risk 1: Stage 5 becomes an encoder cache and stage dynamics collapse. Kill it if stage entropy drops hard, `pi_5` saturates, or routing frequency falls versus CE-only. Main risk 2: encoder and AR gradients conflict on shared trunk parameters. Mitigate with the phased schedule, disjoint token masks, and only then PCGrad on `L_enc` and `L_ar` if needed. Main risk 3: one teacher gets ignored. Track `enc_mask_frac`, `ar_mask_frac`, held-out encoder cosine, and held-out AR KL; if one stays flat with tiny active coverage, your gates are wrong. Main risk 4: `lambda` becomes meaningless. Never supervise `lambda` directly from teacher targets; monitor Spearman correlation between `lambda` and token loss. If that collapses, the KD is overriding the epistemic story.

- **F) Simplest first experiment.** Yes, you can test the idea at `dim=128` on CPU in 30 minutes, but only with sparse cached teacher targets. Use `seq_len=64`, `max_steps=3`, `1024` MiniPile sequences, 2 encoder anchors per sequence at fixed positions like `16` and `48`, and Stage-7 KD only on the final 1-2 positions. Precompute DeBERTa prefix targets and GPT-2 top-32 logits once; do not run teachers online. Run two models for `200` steps: CE-only vs dual-teacher. Metrics: held-out CE/BPT, held-out encoder cosine on anchors, held-out AR KL, stage entropy, `pi_5`/`pi_7` occupancy, and correlation of `lambda` with token loss. Success is not just better BPT; you want small BPT gain plus no collapse in stage entropy and no loss of `lambda` calibration.

- **G) Architecture changes.** Minimal but real. Training loop changes alone are insufficient. You need: `1)` trace outputs from the model for per-step `mu_post_write`, `pi`, `lambda`, logits, and verify state, `2)` a Stage-5 projector head, and `3)` trainer-side teacher caches and masked losses. You do not need new core stages for v1. Parameter overhead is small: a single `768 -> 768` Stage-5 projector is about `0.59M` params, and the optional 2-d gap head is negligible. The bigger architectural change is not parameter count; it is upgrading the current simplified forward in [sutra_v05_ssm.py](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/code/sutra_v05_ssm.py#L267) so Stage 7 is actually part of the recurrent computation instead of a defined-but-unused module at [sutra_v05_ssm.py](/C:/Users/devan/OneDrive/Desktop/Projects/AI%20Moonshots/moonshot-sutra/code/sutra_v05_ssm.py#L154).

Bottom line: the first correct version is sparse, late, gated, and asymmetric. DeBERTa teaches Stage 5 only through prefix-safe anchor states. GPT-2 teaches Stage 7 only on unresolved readouts. `lambda` stays internal. No teacher identity is part of inference. If dual-teacher gain only appears when stage metrics collapse, the idea failed. If gain appears while `pi`, `lambda`, and verify behavior stay real, that is actual absorption.
