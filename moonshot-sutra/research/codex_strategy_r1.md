The most likely outcome is **Scenario C**: Sutra shows genuine strengths on structured and sequential behavior, but the 475M byte-level model lands around **BPB 1.2-1.6**, not near Pythia-class open-text quality. 1. **Most likely outcome:** `C`, about **60%**. My full split is **A 5% / B 25% / C 60% / D 10%**. I would center the forecast at **BPB ~1.35-1.5**.
2. **187x data disadvantage:** fatal for an outright “architecture beats scale” claim on broad language modeling. Not fatal for showing a valuable inductive bias. I would expect architecture to buy **a few-x** sample efficiency if the thesis is real, not **~187x**.
3. **Byte-level vs token-level:** BPB is still the right comparison metric if computed per raw byte. So byte-level does **not** make the metric unfair. What it does do is make the task harder at fixed params/tokens: longer sequences, more steps per concept, worse sample efficiency. So byte-level hurts the model’s ability to reach a good BPB frontier, not the validity of BPB itself.
4. **If BPB = 1.5:** the fastest path to competitive is **not** “train the same byte model longer and hope.” The fastest path is a **token-level Sutra** that keeps the local+sparse thesis, shortens sequence length, and frees capacity for concept-level modeling. Add a tiny global memory mechanism at the same time.
5. **Prepare a token-level fallback?** Yes. Not later, now. Treat it as a parallel branch, not an emergency rewrite.
6. **Single architectural change with highest expected impact:** **add a tiny global scratchpad / 8-16 memory tokens**. That is the cleanest fix for the repo’s recurring failure mode: over-squashing and weak exact retrieval. If you allow representation changes, tokenization may matter even more, but as a pure architecture change the scratchpad is my pick.
